### 1. 접근성 개선 실천: 제품에서 구체적인 노력과 과제

첫 번째 세션에서는 LINE야후의 **웹 액세시빌리티 팀**과 LINE Gift, Yahoo! 재팬 검색 팀이 사용자 접근성 개선을 위한 구체적인 노력을 공유했습니다.

**LINE Gift의 접근 방식**은 **상향식(Bottom-up) 프로젝트**로, 디자인, 개발, QA, 기획 등 다양한 역할의 담당자들이 모여 주도적으로 개선을 추진했다고 합니다. 주요 개선 사항으로는 이미지에 대한 **대체 텍스트(alt text) 누락 방지 시스템 구축**, 하단 탭 UI의 음성 안내 개선 및 시인성 향상을 위한 **명암비 조정**, 그리고 5초 이상 지속되는 애니메이션의 자동 정지 기능 도입 등이 있었습니다. 특히 애니메이션 관련하여 "집중력에 문제가 있는 사용자에게 방해가 될 수 있다"는 점을 공감시키기 위해 일상적인 예시를 들어 소통한 점이 인상 깊었습니다.

**Yahoo! 재팬 검색의 노력**은 레거시 코드 리팩토링 경험에서 시작되었습니다. 디자이너가 직접 코드를 다루는 개발 문화 덕분에 UI/UX와 기술적 구현을 함께 고려한 접근이 가능했습니다. **Lint 도입**을 통한 마크업 검사, `alt` 텍스트의 올바른 사용 유도, 스크린 리더의 읽기 순서 개선, 그리고 재사용 가능한 컴포넌트 및 유틸리티 함수 개발이 핵심이었습니다. 검색 서비스는 사용자 수가 매우 많아 수정사항 배포에 어려움이 있을 수 있지만 꾸준히 개선을 진행하고 있다고 합니다.

**LINE야후 웹 액세시빌리티 팀**은 과거 LINE과 Yahoo!가 각자 가지고 있던 **액세시빌리티 가이드라인을 통합**하고 이를 전사에 보급하는 데 주력했습니다. 가이드라인을 10개로 분할하여 월별 설명회를 개최하는 등 실질적인 이해도를 높이기 위한 노력을 기울였습니다. 궁극적인 목표는 개별 프로덕트가 자율적으로 액세시빌리티를 개선하는 문화를 만드는 것이라고 강조했습니다.

이 세션은 조직적 차이에도 불구하고, **상향식 접근, 꾸준한 개선 노력, 그리고 활발한 지식 공유**가 액세시빌리티 문화 확산에 중요하다는 공통된 메시지를 전달했습니다.

### 2. 숫자가 이끄는 통찰: 웹 성능과 비즈니스 지표의 상관 관계 탐구

두 번째 세션에서는 웹 퍼포먼스와 비즈니스 지표 간의 상관관계를 심도 깊게 분석했습니다.

가장 눈에 띄는 통찰은 **LCP(Largest Contentful Paint)**가 빠를수록 **전환율(CVR)**이 높아지는 경향을 보인다는 점입니다. 특히 **1.5초 이하**의 LCP는 2.5초라는 일반적인 Core Web Vitals 권장 기준보다 훨씬 더 큰 비즈니스 임팩트를 가져온다고 강조했습니다. 이는 단순히 페이지 로딩 속도가 빠르다는 것을 넘어 사용자 경험이 비즈니스 성과에 직접적인 영향을 미친다는 것을 보여줍니다.

또한 **이탈률(Bounce Rate)과 종료율(Exit Rate)**은 페이지 로딩 속도에 강한 역상관관계를 보였습니다. 페이지 로딩이 느릴수록 사용자가 즉시 이탈하거나 페이지를 벗어날 확률이 높아진다는 것입니다. 반면 CLS(Cumulative Layout Shift)와 INP(Interaction to Next Paint)는 CVR과의 상관관계가 비교적 약하게 나타났지만 사용자 경험 측면에서는 여전히 중요하다고 언급했습니다.

이러한 분석을 위해 **최소한의 코드로 구현 가능한 측정 라이브러리**를 제공하여 쉽게 데이터를 수집하고 Tableau를 활용한 대시보드로 시각화하며 웹 퍼포먼스 팀이 기술 컨설팅을 지원하는 **전사적 지원 체계**를 구축했습니다.

성능 개선을 위한 팁으로는 **Critical Rendering Path 이해, HTML 파싱 방해 요소 제거 (preload, async/defer 사용), CSS/JS 최적화** 등 프론트엔드 기술과 함께, **빠른 백엔드 응답 시간(TTFB), 컴퓨팅 부하 절감, CPU 프로파일링 활용** 등 백엔드 최적화의 중요성을 강조했습니다. 마지막으로 향후에는 **LLM과 소스 코드를 활용하여 AI 기반의 성능 최적화 제안**을 자동화하는 연구를 진행 중이라고 밝혀 기대를 모았습니다.

저도 SEO팀에서 일할 때 LCP와 CLS를 개선하는 작업을 해봤었기에 더 이해가 잘 되고 공감가는 세션이었습니다.

### 3. "Yahoo! JAPAN Search"의 웹 성능 개선 노력

세 번째 세션에서는 Yahoo! 재팬 검색팀의 Core Web Vitals를 활용한 웹 퍼포먼스 개선 사례를 자세히 공유했습니다. Yahoo! 재팬 검색은 쿼리에 따라 화면에 표시되는 내용이 다양하게 변화하는 특성 때문에 퍼포먼스 개선에 더욱 복잡한 과제를 안고 있다고 합니다.

![Core Web Vitals](/posts/inspiration/techverse2025/coreVitals.avif)
Yahoo! 재팬 검색 팀은 **LCP(2.5초 미만 권장), INP(200ms 미만 권장), CLS(0.1 미만 권장)**와 같은 Core Web Vitals 지표를 통해 정량적으로 성능을 측정하는데 이러한 지표 개선은 단순히 SEO를 넘어 **사용자 경험 향상**에 중점을 두며 실제로 성능 수치가 좋아지면 클릭 수 및 노출 수가 증가하고 사용자가 클릭하지 않는 비율이 감소하는 경향을 확인했다고 합니다.

성능 시각화를 위해 **랩 데이터(Lab data)**와 **필드 데이터(Field data)**를 모두 활용하고 랩 데이터는 통제된 환경에서 대표적인 쿼리에 대해 측정하여 개선 지점을 파악하는 데 유용하며 필드 데이터는 실제 사용자 브라우저에서 수집되어 실제 성능 상황을 파악하는 데 사용된다고 합니다. 또 대시보드를 통해 쿼리 영역별 성능 추이를 파악하고 문제가 있는 영역을 특정하여 HTML 요소 레벨까지 분석한다고 합니다.

구체적인 개선 사례로는 인물 정보 영역에서 공통적으로 발생하는 레이아웃 시프트를 특정하고 Lighthouse 툴로 확인하여 수정했고 지역 정보 쿼리에서 LCP를 저하시키는 큰 이미지 요소를 파악하여 최적화 했다고 합니다.

### 4. Next.js App Router로의 마이그레이션 및 K8s에서 찾은 최상의 솔루션

마지막 세션에서는 LINE Plus 콘텐츠 서비스 웹 팀의 정규성 님과 조정근 님이 Open Chat 서비스의 Next.js App Router 전환 및 Kubernetes(K8s) 도입 경험을 공유했습니다. Open Chat은 일본, 태국, 대만에서 서비스되는 익명 채팅 서비스로 LINE 본체뿐만 아니라 여러 웹 서비스로 구성됩니다.

![Multi Tenancy](/posts/inspiration/techverse2025/multiTenancy.avif)
기존 Jenkins 기반 배포 방식의 한계(서비스별 서버 구성, 환경 관리의 어려움)를 극복하기 위해 **Kubernetes의 멀티 테넌시(Multi-Tenancy) 전략**을 선택했다고 합니다. 이는 하나의 클러스터 내에 네임스페이스를 통해 리소스를 논리적으로 분리하여, 자원 격리, 트래픽 분리, 운영 및 모니터링 효율성을 높이는 방식이고 `Phase`와 `Stage` 개념을 도입하여 환경 및 버전 관리를 세밀하게 할 수 있다고 합니다. 배포 도구로는 **Helm Chart**와 **ArgoCD**를 활용했다고 합니다.

![what Nextjs](/posts/inspiration/techverse2025/nextjs.avif)
그리고 Next.js App Router로의 마이그레이션은 FCP와 LCP 개선을 통한 사용자 체감 성능 향상을 목표로 했다고 합니다. 그래서 기존의 CSR 처리를 Hybrid CSR과 SSR로 변경했다고 합니다. 실제 마이그레이션 결과 **FCP는 61%, LCP는 78% 개선**되는 극적인 효과를 얻었다고 합니다.
배포 시에는 Docker 컨테이너와 **Standalone Build**를 활용하여 필요한 의존성만 추출하고 정적 파일은 CDN에 올려 이미지 크기를 94%(`10GB`에서 `600MB`) 감소시키는 데 성공했다고 합니다.

마이그레이션 과정에서 겪었던 주요 이슈와 해결책도 공유했습니다:
*   **페이지 전환 애니메이션 문제**: 코드 스플리팅으로 인한 전환 지연. **해결책**: `Link` 컴포넌트의 `prefetch` 기능을 적극 활용하여 미리 페이지를 로드.
*   **탭 UI 상태 유지 문제**: 탭 전환 시 서버 측 API 호출 반복 및 스크롤 복원 문제. **해결책**: 최초 진입 시에만 SSR을 하고 이후 탭 변경은 클라이언트 측 렌더링(CSR)으로 처리하며, `router.replace` 대신 Native History API를 사용하여 페이지 리로딩 없이 URL을 교체.
*   **Server Action 버전 문제**: 리팩토링이나 빌드 환경 변경 시 Server Action ID가 변경되어 발생하는 오류. **해결책**: 배포 시 이전 버전의 Server Action을 유지하고, `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY` 환경 변수를 설정하여 빌드 간 일관된 ID를 유지하며 추후 Sticky Session 방식 도입을 고려.

향후에는 멀티 컨테이너 K8s 환경에서 인메모리 캐시 분산 문제를 해결하기 위해 **Redis와 같은 외부 저장소에 캐시를 통합**하는 방안을 모색 중이라고 밝혔습니다.

아무래도 Next.js가 주제 다 보니 가장 이해가 잘 가는 발표였습니다. 현재 일하는 곳에서도 Docker + AWS로 서비스를 하다보니 Standalone Build로 전환해서 이미지 사이즈를 줄이는 방식을 도입했었는데 같은 내용이 나와 놀라웠습니다.
또 prefetch를 활용하는 것은 예전 SEO팀에서도 사용해보았기에 발표를 들으면서 저도 여러가지 시행착오를 겪었던 기억이 떠올랐습니다.
