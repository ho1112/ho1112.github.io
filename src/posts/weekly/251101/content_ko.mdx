## 🔥 이번 주 핵심 테마
이번 주는 네이티브 CSS의 발전과 React 생태계의 주요 업데이트가 중심을 이뤘습니다. CSS는 `linear()` 타이밍 함수와 앵커 포지셔닝(Anchor Positioning) 같은 새로운 기능으로 복잡한 애니메이션과 UI를 자바스크립트 없이 구현할 수 있는 가능성을 열었습니다. 동시에 React 진영에서는 React Compiler 1.0, Next.js 16 Beta, Bun 1.3 등 핵심 도구들이 대거 릴리스되며 개발 환경의 큰 변화를 예고했습니다.

## 1. 🔍 심층 분석

- **[Springs and Bounces in Native CSS](https://frontendfoc.us/link/176210/web)** (출처: *Frontend Focus #715*)
    ![Springs and Bounces in Native CSS](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/v1761736308/fg4ro1px2ykwcqnnnppo.png)
    - **배경**: 전통적으로 스프링(spring)이나 바운스(bounce) 같은 물리 기반 애니메이션은 JavaScript 라이브러리의 영역이었습니다. 하지만 CSS `linear()` 타이밍 함수가 등장하면서 네이티브 CSS만으로도 이러한 정교한 효과를 구현할 수 있게 되었습니다.
    - **핵심 기능**: Josh W. Comeau가 `linear()` 함수를 사용하여 스프링 및 바운스 효과를 만드는 접근 방식을 탐구합니다. 그는 "일반적인 타이밍 함수 몇 개를 전역 CSS 변수에 저장하는" 방식을 통해 지금 당장 이 기능들을 가장 잘 활용할 수 있는 팁을 공유합니다.
    - **실무 적용**: `linear()` 함수는 `transition-timing-function` 이나 `animation-timing-function` 속성에서 복잡한 Easing 곡선을 정의할 수 있게 해줍니다.
    - **참고**: `linear()` 타이밍 함수(사전 설정된 `linear` 값과는 다름)의 브라우저 지원 범위는 현재 약 88%입니다. (출처: *Frontend Focus #715*)

- **[CSS Anchor Positioning](https://ishadeed.com/article/anchor-positioning/)** (출처: *Smashing Newsletter #531*)
    ![The Basics Of Anchor Positioning](https://mcusercontent.com/16b832d9ad4b28edf261f34df/images/337786aa-6ab7-b4f3-52f5-3818c50aac8a.png)
    - **배경**: 특정 요소를 다른 요소(부모가 아닐 수도 있음)에 상대적으로 배치하는 것은 CSS에서 까다로운 작업이었습니다. 예를 들어, 카드 내의 태그를 이미지의 특정 모서리에 고정시키려 할 때, 카드 레이아웃이 변경되면(예: 데스크톱에서 수평 정렬) 배치가 깨지기 쉬웠습니다.
    - **핵심 기능**: 앵커 포지셔닝(Anchor Positioning) API는 한 요소가 다른 '앵커' 요소에 상대적으로 위치를 잡을 수 있도록 허용합니다. 이는 두 요소가 DOM 트리 상에서 부모-자식 관계가 아니더라도 가능합니다.
    - **실무 적용**: Ahmad Shadeed는 이 기능을 사용하여 툴팁이나 팝오버를 특정 UI 요소에 연결하는 방법을 소개합니다. 특히 `position-try-fallbacks` 속성을 사용하면, 대상 요소가 뷰포트에 맞지 않을 경우 자동으로 위치를 전환(예: 툴팁을 위에서 아래로)하도록 설정할 수 있어, 깨지기 쉬운 핵(hack) 없이도 훨씬 강력한 UI 구현이 가능해집니다.

## 3. 📝 추천 아티클 & 팁

    + **핵심 인사이트**: 모바일 기기를 타겟으로 할 때는 작은 번들 크기와 빠른 렌더링 시간이 핵심입니다. 이 글은 동일한 앱을 Marko, SolidStart, SvelteKit, Qwik, Nuxt, Next.js 등 10가지 다른 접근 방식으로 구축하여 모바일 성능을 비교 분석합니다. 어떤 프레임워크가 모바일 환경에서 뛰어난 성능을 보이는지 실제 데이터를 통해 확인할 수 있습니다.

- **[Compressed Fluid Typography](https://matthiasott.com/notes/compressed-fluid-typography)** (출처: *Smashing Newsletter #531*)
    ![Compressed Fluid Typography example](https://mcusercontent.com/16b832d9ad4b28edf261f34df/images/cf485292-226d-79ce-3717-fa59913c68be.jpg)
    + **핵심 인사이트**: 유동적인 타이포그래피(Fluid Typography) 계산은 보통 기본 폰트 크기를 16px로 가정합니다. 하지만 사용자가 브라우저의 기본 폰트 크기를 24px로 변경하면 어떻게 될까요? Matthias Ott는 사용자의 기본 폰트 크기가 커질수록 타이포그래피의 전반적인 증가율을 늦추는 '감쇠 계수(damping factor)'를 계산에 도입하는 접근 방식을 공유합니다. 이를 통해 사용자의 선호도를 존중하면서도 폰트 스케일이 제어 불가능하게 커지는 것을 방지할 수 있습니다.

- **[Inlining Critical CSS: Does It Make Your Website Faster?](https://www.debugbear.com/blog/critical-css?utm_source=CSS-Weekly&utm_campaign=Issue-625&utm_medium=web)** (출처: *CSS Weekly #625*)
    + **핵심 인사이트**: 'Critical CSS'는 페이지의 스크롤 없이 볼 수 있는 부분(above-the-fold)을 렌더링하는 데 필요한 최소한의 CSS를 의미합니다. Matt Zeunert는 이 Critical CSS가 무엇인지, 이를 인라인(inline)으로 삽입하여 페이지 로드 성능(특히 FCP, LCP)을 개선하는 방법과, 이로 인해 발생하는 트레이드오프(캐싱 문제 등)에 대해 설명합니다.

- **[Why `NaN !== NaN` in JavaScript (and the IEEE 754 Story Behind It)](https://javascriptweekly.com/link/176452/web)** (출처: *JavaScript Weekly #759*)
    + **요약**: `NaN`은 숫자(number) 타입이지만, 동시에 '숫자가 아님(Not a Number)'을 의미합니다. 이로 인해 `NaN === NaN`이 `false`가 되는 잘 알려진 동작이 발생합니다. 이 글은 이것이 JavaScript의 버그가 아니라, 부동소수점 표준인 IEEE 754에서 비롯된 이유를 설명합니다.

## ✍️ 마치며
이번 주 소식은 React 생태계의 대대적인 도구(Compiler, Next.js, Bun) 업데이트와 함께, CSS가 얼마나 정교한 수준까지 발전했는지(애니메이션, 앵커 포지셔닝)를 동시에 보여주고 있습니다. 특히 React Compiler 1.0의 등장은 React 개발의 패러다임을 또 한 번 바꿀 수 있는 잠재력을 지니고 있어, 앞으로의 성능 최적화 추이를 지켜볼 필요가 있겠습니다. 네이티브 웹 플랫폼의 기능이 강력해지는 만큼, 이를 활용하는 프레임워크와 도구들도 함께 진화하는 한 주였습니다.
