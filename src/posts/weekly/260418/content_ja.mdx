## 🔥 今週の主要テーマ
今週はフロントエンドエコシステムの主要なレンダリングアーキテクチャの変化とWeb Componentsの活用事例が注目を集めました。MDNがReactを取り除き、Web Componentsベースに新しく転換した事例や、TanStack Startが考案した新しい形のReact Server Components(RSC)サポート方式が目を引きます。さらに、UI開発の生産性を高めるスケルトン自動生成ツールや軽量MLライブラリ、そしてアクセシビリティやタイポグラフィについての深みのある議論がありました。

## 1. 🔍 深掘り
- **[Under the Hood of MDN's New Frontend](https://developer.mozilla.org)** (出典: *JavaScript Weekly #781* )
	![MDN Web Docs](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/gm0i5twsmyfutkc1xygg.jpg)
	- 背景: MDNは既存のCreate React AppベースのSPAスタックが、静的ドキュメント中心のコンテンツと根本的に合わず、技術的負債と絡み合ったCSS問題に悩まされていました。
	- コア機能: Reactを排除し、LitベースのWeb Componentsと独自開発したサーバーコンポーネントシステムを採用してアーキテクチャを完全に再構築しました。MarkdownをJSONに変換した後、SSRを経て不要なJavaScriptの転送を最小限に抑えます。
	- コード例:
		```javascript
		// Litベースの軽量なWeb Components構造の例（MDNアーキテクチャ参考）
		import {html, css, LitElement} from 'lit';

		class MdnDocSnippet extends LitElement {
			static styles = css`p { color: blue }`;
			render() {
				return html`<p>MDN Web Components レンダリングの例です。</p>`;
			}
		}
		customElements.define('mdn-doc-snippet', MdnDocSnippet);
		```
	- 実務への適用: 重いJavaScriptフレームワークを削ぎ落とし、パフォーマンスと読み込み速度を最適化する必要があるコンテンツ主体のプラットフォームに適しています。
	- 注意事項: 既存の豊富なReactエコシステムやサードパーティライブラリを放棄し、純粋なWebプラットフォームAPI（Vanilla JS/HTML/CSS）により近い形で開発しなければならない学習コストが存在します。
- **[TanStack Start Adds React Server Components Support](https://tanstack.com/blog/react-server-components)** (出典: *React Status #470* )
	![TanStack Start Adds React Server Components Support](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/yfbmmkgn28b10zyjypzm.jpg)
	- 背景: フルスタックフレームワークであるTanStack Startが、Next.jsとは全く異なる「Isomorphic-first（同型優先）」のアプローチと「RSCはデータである」という哲学を加えてRSCサポートを追加しました。
	- コア機能: 意図的にセキュリティ脆弱性の原因となり得る曖昧な`use server`ディレクティブを排除しました。代わりに`createServerFn`を通じてサーバー境界を明確にし、サーバーが開けておいたスロットをクライアントが埋めることができるComposite Components機能を提供します。
	- コード例:
		```javascript
		import { createServerFn } from '@tanstack/react-start'

		// 'use server'の代わりに明示的なサーバー関数宣言で境界を作成
		const fetchUserData = createServerFn().handler(async () => {
			// サーバー専用ロジック（RSCをデータとして返却可能）
			const data = await db.query('...');
			return { user: data };
		});
		```
	- 実務への適用: 既存のNext.jsなどとは異なる方式のRSCルーティングやデータフェッチング構造が必要な、Reactベースのフルスタックアプリケーション開発に活用できます。
	- 注意事項: フレームワーク自体の設計思想が既存のサーバーファースト（Server-first）RSCとは異なるため、クライアント中心（Client-first）のメンタルモデルへの転換が必要です。

## 2. 🛠️ ツール & ライブラリ
- **[Micro-ML: A Toolkit of Forecasting and Clustering Algorithms](https://javascriptweekly.com/link/183719/web)** (出典: *JavaScript Weekly #781* )
	![Micro-ML: A Toolkit of Forecasting and Clustering Algorithms](https://raw.githubusercontent.com/AdamPerlinski/micro-ml/main/docs/logo.svg)
	+ 概要: TensorFlow.jsのような重いライブラリを導入しなくても、回帰、スムージング、クラスタリング、データ分類、および時系列予測を実行できる約56KBの容量のWASMベースMLライブラリです。
	+ 使い方:
		```javascript
		import { regression } from 'micro-ml';

		// 重いTensorFlowなしでブラウザ内で軽く時系列予測を実行
		const model = regression.linear(historicalData);
		const prediction = model.predict(nextTimeStep);
		```
	+ 実践チップ: クライアントサイドでリアルタイムのデータ予測が必要なダッシュボードや軽量な分析ツールの開発時に、ロードタイムとバンドルサイズを劇的に節約できます。

- **[Boneyard: Auto-Generated Skeleton Screens for Your UI](https://javascriptweekly.com/link/183718/web)** (出典: *JavaScript Weekly #781* )
	![Boneyard: Auto-Generated Skeleton Screens for Your UI](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/afxlfwdqfincfdil92bq.jpg)
	+ 概要: 実際のUIのスナップショットを撮り、正確な位置とサイズを持つ四角形の「骨組み（bones）」リストをキャプチャしてスケルトンスクリーンを自動生成するツールです。
	+ 使い方:
		```javascript
		import { Boneyard } from 'boneyard';

		// 現在レンダリングされているコンテナのレイアウトに基づいてスケルトンを自動キャプチャ
		const skeletonUI = Boneyard.generateSkeleton(document.getElementById('content'));
		```
	+ 実践チップ: React、Preact、React Native、Vue、Svelte、Angularをすべてサポートしているため、レイアウトのスケルトンを手動でCSSに合わせて調整する手間を省いてくれます。

## 3. 📝 おすすめ記事 & チップ

- **[You Can't Cancel a Promise (Except Sometimes You Can)](https://javascriptweekly.com/link/183710/web)** (出典: *JavaScript Weekly #781* )
	![You Can't Cancel a Promise](https://www.inngest.com/_next/image?url=%2Fassets%2Fblog%2Fhanging-promises-for-control-flow%2Ffeatured-image.jpg&w=1920&q=75)
	+ 主要なインサイト: JavaScriptは本質的に、実行中の`Promise`を強制的にキャンセル(Cancel)する組み込みAPIを持っていません。しかし著者は、**「絶対にresolveやrejectされない（永遠にPending状態である）Promise」**を作成し、それを`await`させることで、該当の非同期関数(async function)の実行をその場で永久に停止させてしまうテクニックを紹介しています。
	+ 動作原理: 関数が停止しているとメモリリークが発生しそうですが、該当のPromiseを参照する外部コードがなければ、V8エンジンのガベージコレクタ(GC)が停止してしまった関数コンテキストを静かにメモリから整理してくれます。
	+ コード例:
		```javascript
		// 絶対に終わらないPromiseを生成
		const neverResolve = new Promise(() => {});

		async function myLongWorkflow() {
			console.log("作業開始...");
			// 割り込み発生時、永遠に待機させる
			await neverResolve;
			// この下のコードは永遠に実行されず、その後GCによって関数コンテキストが整理される
			console.log("このコードは絶対に実行されません。");
		}
		```
	+ 実務での活用: Inngestのようなエンジンでは、このトリックを通じてサーバーレス環境で実行中の長く複雑な非同期ワークフローを安全に強制中断させるために積極的に活用しています。
- **[The Vertical Codebase](https://tkdodo.eu/blog/the-vertical-codebase)** (出典: *React Status #470* )
	![The Vertical Codebase](https://tkdodo.eu/blog/_astro/GEg89DDXsAA8DC.DbrOSQsh.jpeg)
	+ 主要なインサイト: `components/`、`hooks/`、`utils/`の形の水平的なフォルダ構造は、初期には綺麗に見えますが、時間が経つにつれてメンテナンスが難しくなります。著者はSentryのコードベースの事例をもとに、ドメイン優先の垂直的(Vertical)な構造を採用すべきだと主張しています。
- **[Dyslexia-Friendly Fonts](https://pimpmytype.com/dyslexia-fonts/)** (出典: *Smashing Newsletter #554* )
	![Dyslexia-Friendly Fonts](https://mcusercontent.com/16b832d9ad4b28edf261f34df/images/89dffee4-31cf-bc6f-6e35-aaa7212c8cf5.png)

## ✍️ おわりに
毎週新しく溢れ出るツールやアーキテクチャの変遷を見ていると、Webエコシステムが絶えず「削ぎ落とし」と「構造化」の間でバランスを見出していることを感じます。MDNが古いフレームワークスタックを取り除きWeb Componentsに回帰した点や、TanStack Startが既存のServer Componentsの軌道をねじ曲げて明示的な境界を開拓する姿は特に印象深いです。銀の弾丸はないという事実をもう一度実感し、単なる流行よりも言語や技術の動作原理（Promiseの中断トリックなど）を深く理解し、それぞれのドメインに合った適正な技術を悩む一週間になればと思います。
