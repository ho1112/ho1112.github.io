## 🔥 이번 주 핵심 테마
이번 주에는 프론트엔드 생태계의 주요 렌더링 아키텍처 변화와 웹 컴포넌트 활용 사례가 주목을 받았습니다. MDN이 React를 걷어내고 웹 컴포넌트 기반으로 새롭게 전환한 사례와, TanStack Start가 고안한 새로운 형태의 React Server Components(RSC) 지원 방식이 눈길을 끕니다. 더불어 UI 개발 생산성을 높여주는 스켈레톤 자동 생성 도구 및 경량 ML 라이브러리, 그리고 접근성 및 타이포그래피에 대한 심도 있는 논의가 있었습니다.

## 1. 🔍 심층 분석
- **[Under the Hood of MDN's New Frontend](https://developer.mozilla.org)** (출처: *JavaScript Weekly #781* )
  ![MDN Web Docs](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/gm0i5twsmyfutkc1xygg.jpg)
  - 배경: MDN은 기존 Create React App 기반의 SPA 스택이 정적 문서 중심의 콘텐츠와 근본적으로 맞지 않아 기술 부채와 얽힌 CSS 문제를 겪고 있었습니다.
  - 핵심 기능: React를 제거하고 Lit 기반의 웹 컴포넌트(Web Components)와 자체 개발한 서버 컴포넌트 시스템을 채택하여 아키텍처를 완전히 재구축했습니다. Markdown을 JSON으로 변환한 뒤 SSR을 거쳐 불필요한 자바스크립트 전송을 최소화합니다.
  - 코드 예제:
    ```javascript
    // Lit 기반의 가벼운 웹 컴포넌트 구조 예시 (MDN 아키텍처 참고)
    import {html, css, LitElement} from 'lit';

    class MdnDocSnippet extends LitElement {
      static styles = css`p { color: blue }`;
      render() {
        return html`<p>MDN 웹 컴포넌트 렌더링 예시입니다.</p>`;
      }
    }
    customElements.define('mdn-doc-snippet', MdnDocSnippet);
    ```
  - 실무 적용: 무거운 자바스크립트 프레임워크를 덜어내고 성능 및 로딩 속도를 최적화해야 하는 콘텐츠 위주의 플랫폼에 적합합니다.
  - 주의사항: 기존의 풍부한 React 생태계와 서드파티 라이브러리를 포기하고, 순수 웹 플랫폼 API(Vanilla JS/HTML/CSS)에 더 가깝게 개발해야 하는 러닝 커브가 존재합니다.
- **[TanStack Start Adds React Server Components Support](https://tanstack.com/blog/react-server-components)** (출처: *React Status #470* )
  ![TanStack Start Adds React Server Components Support](https://res.cloudinary.com/cpress/image/upload/w_1280,e_sharpen:60,q_auto/yfbmmkgn28b10zyjypzm.jpg)
  - 배경: 풀스택 프레임워크인 TanStack Start가 Next.js와는 전혀 다른 'Isomorphic-first(동형 우선)' 접근법과 "RSC는 데이터다"라는 철학을 가미하여 RSC 지원을 추가했습니다.
  - 핵심 기능: 의도적으로 보안 취약점의 원인이 될 수 있는 모호한 `use server` 지시어를 배제했습니다. 대신 `createServerFn`을 통해 서버 경계를 명확히 하고, 서버가 열어둔 슬롯을 클라이언트가 채울 수 있는 Composite Components 기능을 제공합니다.
  - 코드 예제:
    ```javascript
    import { createServerFn } from '@tanstack/react-start'

    // 'use server' 대신 명시적인 서버 함수 선언으로 경계 생성
    const fetchUserData = createServerFn().handler(async () => {
      // 서버 전용 로직 (RSC를 데이터처럼 반환 가능)
      const data = await db.query('...');
      return { user: data };
    });
    ```
  - 실무 적용: 기존의 Next.js 등과 다른 방식의 RSC 라우팅 및 데이터 페칭 구조가 필요한 React 기반 풀스택 애플리케이션 개발에 활용할 수 있습니다.
  - 주의사항: 프레임워크 자체의 설계 사상이 기존의 서버-퍼스트(Server-first) RSC와 다르기 때문에, 클라이언트 중심(Client-first)의 멘탈 모델로의 전환이 필요합니다.

## 3. 📝 추천 아티클 & 팁

- **[You Can't Cancel a Promise (Except Sometimes You Can)](https://javascriptweekly.com/link/183710/web)** (출처: *JavaScript Weekly #781* )
  ![You Can't Cancel a Promise](https://www.inngest.com/_next/image?url=%2Fassets%2Fblog%2Fhanging-promises-for-control-flow%2Ffeatured-image.jpg&w=1920&q=75)
  + 핵심 인사이트: JavaScript는 태생적으로 실행 중인 `Promise`를 강제로 취소(Cancel)하는 내장 API가 없습니다. 하지만 저자는 **"절대 resolve나 reject가 되지 않는(영원히 Pending 상태인) Promise"** 를 만들어 이를 `await` 하게 하면, 해당 비동기 함수(async function)의 실행을 그 자리에 영구적으로 멈춰버릴 수 있는 기법을 소개합니다.
  + 동작 원리: 함수가 멈춰있으면 메모리 누수가 생길 것 같지만, 해당 Promise를 참조하는 외부 코드가 없다면 V8 엔진의 가비지 컬렉터(GC)가 멈춰버린 함수 컨텍스트를 조용히 메모리에서 정리해 버립니다.
  + 코드 예제:
    ```javascript
    // 절대 끝나지 않는 Promise 생성
    const neverResolve = new Promise(() => {});

    async function myLongWorkflow() {
      console.log("작업 시작...");
      // 인터럽트 발생 시, 영원히 대기하게 만듦
      await neverResolve;
      // 이 아래 코드는 영원히 실행되지 않으며, 이후 GC에 의해 함수 컨텍스트가 정리됨
      console.log("이 코드는 절대 실행되지 않습니다.");
    }
    ```
  + 실무 활용: Inngest와 같은 엔진에서는 이 트릭을 통해 서버리스 환경에서 실행 중인 길고 복잡한 비동기 워크플로우를 안전하게 강제 중단시키는 데 적극적으로 활용합니다.
- **[The Vertical Codebase](https://tkdodo.eu/blog/the-vertical-codebase)** (출처: *React Status #470* )
  ![The Vertical Codebase](https://tkdodo.eu/blog/_astro/GEg89DDXsAA8DC.DbrOSQsh.jpeg)
  + 핵심 인사이트: `components/`, `hooks/`, `utils/` 형태의 수평적 폴더 구조는 초기에는 깔끔해 보이지만 시간이 지날수록 유지보수하기 어려워집니다. 저자는 Sentry의 코드베이스 사례를 바탕으로 도메인 우선의 수직적(Vertical) 구조를 채택해야 한다고 주장합니다.

MDN이 낡은 프레임워크 스택을 덜어내고 웹 컴포넌트로 회귀한 점이나, TanStack Start가 기존 Server Components의 궤도를 비틀어 명시적인 경계를 개척하는 모습은 특히 인상 깊습니다. 은총알은 없다는 사실을 또 한 번 실감하며, 단순한 유행보다는 언어와 기술의 동작 원리(Promise 중단 트릭 등)를 깊이 이해하여 각자의 도메인에 맞는 적정 기술을 고민하는 한 주가 되었으면 합니다.
