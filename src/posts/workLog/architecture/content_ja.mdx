## はじめに：3つのプロジェクトを振り返って

これまでフロントエンドエンジニアとして参加した3つのプロジェクト（ホテル予約プラットフォーム、ウェブ小説公募展サイト、オファー型採用プラットフォーム）のコードベースを振り返り、プロジェクトの複雑度を管理するためにコンポーネントの役割を分け、構造を設計した過程をまとめました。
アーキテクチャの理論よりも、当時のサービスの状況や技術的な制約の中で、関心の分離を実務環境に合わせてどのように実装するか、チームメンバーと一緒に悩んだ記録です。
当時に作成した開発メモや個人的な記憶をもとにまとめているため、実際のプロダクションコードの詳細な名称やディレクトリ構造とは異なる場合があります。記憶が曖昧な部分は、当時直面した問題とそれを解決した方法を中心に、大まかな構造と文脈を再構成して作成しました。

## OTAサービス - 検索・予約・SEOが1つのrepoに共存する際のフォルダ規則

### このプロジェクトは？

| 項目 | 内容 |
|------|------|
| **サービス** | OTA(Online Travel Agency)プラットフォーム — 国内・海外ホテル検索・予約 |
| **スタック** | Next.js 12 (Pages Router), React 17, Redux + thunk + redux-form, styled-components, Express カスタムサーバー |
| **役割** | 機能開発・構造改善・**SEOリポジトリ統合の提案・主導**に参加。`uiParts`・`*.page.tsx`などのフォルダ規則は**チームでの相談**により決定 |

### ディレクトリ構造

プロダクションは`***/hotel/`の下にデプロイされ、コード・ビルド・静的アセットも`/hotel`ネームスペースを前提として組まれています。

```text
. (Project Root)
├── server.js                 # Expressサーバー ("/hotel" ネームスペースのサービング、エラールーティング制御)
├── next.config.js            # pageExtensions, redirects/rewrites, Cache-Control, polyfill
├── typings/                  # グローバル型定義 + 外部APIスキーマ(契約)リポジトリ
├── src/                      # アプリケーションのメインロジック
│   ├── pages/                # Next Pages Routerエントリ (*.page.tsxのみをページとして扱う)
│   ├── actions/              # Redux Action/Thunk (非同期、ポーリング、API呼び出しなど)
│   ├── reducers/             # Redux Reducer (状態 + ドメインルールが厚く組み込まれる傾向)
│   ├── store/                # Redux store + redux-persist(session)構成
│   ├── containers/           # 予約/決済などの大きなフローをまとめる連結レイヤー
│   ├── components/           # Atomic DesignベースのUIレイヤー
│   │   ├── atoms/            # 最小UI単位
│   │   ├── molecules/        # 組み合わせUI単位
│   │   ├── organisms/        # ドメインの文脈を持つ大きなUIの塊
│   │   └── uiParts/          # コンテンツ・URLベースのUI
│   ├── features/             # 特定のドメイン(キャンペーンなど)単位でまとめられた機能パッケージ
│   ├── hooks/                # カスタムフック(状態/DOM/ナビゲーション補助)
│   ├── utils/                # envフラグ(getEnvInfo)、クエリ/日付/画像などの共通ユーティリティ
│   ├── constants/            # 定数(環境/SEO/画像パス/デフォルト値など)
│   └── styles/               # グローバルスタイル/テーマ
```

**主なレイヤー別の責任と役割**

| 領域 | 役割 |
|------|---------|
| `*.page.tsx` | URL・`getServerSideProps`・`envInfo` — **ルーティング・データの入り口** |
| `atoms` ~ `organisms` | 検索・詳細・予約・決済 — **Redux・APIと深く結びついたアプリ本体** |
| `uiParts` | TOP・initial・review・SEOエリア — **props・URL中心のセクションUI** |
| `utils/seo` | エリア/温泉/initialなど **SEO URL用のサーバーデータ組み立て** |

### Story 1. uiPartsと*.page.tsx - なぜレイヤーをもう1つ置いたのか

#### 背景: repoの中に「複数の種類の画面」がすでに共存していた

ホテルrepoは最初から**検索・予約アプリ1つ**だけを含んでいた形ではありませんでした。

- **TOP** (`/hotel/`) — エリアリンク、キャンペーン、レビュー要約
- **検索結果・詳細・予約・決済** — Redux・API中心
- **SEO・コンテンツURL** — かつては別のリポジトリで開発・デプロイされ、同じ`/hotel`ドメインでデプロイ
- **initial・review・温泉・都道府県** — SEO専用ページ(URLパターンがそれぞれ異なる)

1つのコードベースの中に **「アプリのフロー」**と**「読んでリンクをたどるページ」** がすでに混ざっており、その後SEOを**完全に1つのrepoに統合する作業**が続きました。

#### 当時チームが実際に不便だったこと

##### 1) Atomicフォルダだけでは「どこに入れるか？」がよく合わなかった

`atoms` / `molecules` / `organisms`は**サイズ**の基準に近いです。しかし実務ではこのような質問が繰り返されました。

- `HotelListLink` — ボタンatomか、TOP用のリンクブロックか？
- `HotelPaging` — moleculeか、SEO・initial用URLページのナビか？
- FAQ + タイトル + 余白 — organismか、「ページセクション」か？

サイズでは分けられても、「検索・予約に属するのか / コンテンツページに属するのか」はフォルダを見ただけでは区別できませんでした。
`uiParts`を**チームで相談して**置いたのも、この**配置の曖昧さ**を減らそうとする目的に近いです。

##### 2) `organisms`が「アプリ本体 + α」として肥大化していた

`Template`、`SearchBox`、`HotelDetail`のような**本当のorganism**の横に、TOPエリア・レビューのページング・(移行中の)SEO FAQ・Swiperのように**Reduxフローとは無関係なUI**も同じ階層に混ざりやすかったです。
レビューのたびに「connect必要？」「検索APIの型を使うべき？」のような**認知負荷**が大きくなり、**organisms = 検索・予約ドメイン**という感覚が薄れていくのが、構造に手を入れることになった直接的な不便さでした。

##### 3) SEO移行を控え、「同じ役割のUIが2セット」あることが判明した

別のSEO repo・過去のコピー版のせいで**似たようなUIがすでに2つの分岐**になっていました。

| 役割 | 既存のホテルアプリ | SEO・コンテンツ側 |
|------|-------------|---------------|
| ホテルカード | `atoms/HotelItem` (Redux, API v3) | 移行時 `HotelItem` (異なる型) |
| ページング | `HotelPaging` (URL移動) | SEOリストもURLページ単位 |
| リンク | `HotelListLink` | エリア・一覧リンク |

organisms/atomsに**そのまま追加すると**名前の衝突・importの混乱・レビュー範囲が一気に大きくなります。移行直後の`HotelItem` → `SwiperHotelItem`への名前変更も、Atomicの再分類ではなく、重複するUIを1つの境界内で整理した選択に近いです。

##### 4) ページファイルが増え、「ルートなのか、画面なのか」の区別が難しくなった

`pages/hotel/`の下に`getServerSideProps`だけがあるファイルと数百行のUI組み立てファイルが**同じディレクトリ・似たような名前**で混在しました。**ルーティング・データの入り口**と**画面の実装**をファイル名だけで区別したいというニーズが`pageExtensions` + `*.page.tsx`へと繋がりました。

#### チームが選んだ方向

**Atomicを教科書通りに再定義しよう** — 移行・スプリントの日程上、現実的ではありませんでした。

代わりに**役割が異なるUIはフォルダでもう一度区別**しようという方向でした。

| レイヤー | 配置の基準 |
|--------|----------------|
| `atoms` ~ `organisms` | 検索・詳細・予約・決済 — **Redux・API・アプリフロー** |
| `uiParts` | TOP・initial・review・SEOエリア — **props・URL・縦セクション** (atoms/organismsの**サイズとは無関係**) |
| `*.page.tsx` | **ルート・SSR・envInfo** |
| 同じフォルダの`index.tsx`など | **画面の組み立て** |

##### `uiParts`に置くことにした理由
1. **Redux・検索状態に縛られない** — `HotelListLink`, `HotelPaging` など
2. **TOP / initial / review / SEO エリアで共通**して使う
3. **ページセクション単位** — `LargeTitle` + FAQ, Swiper ホテル一覧 など
4. **移行・重複時の境界** — SEO UIと既存UIが被る時、片方に集める

#### `pageExtensions` — なぜ `*.page.tsx`なのか

```js title="next.config.js"
pageExtensions: ['page.tsx', 'page.ts'],
```
- **`index.page.tsx`**: URL, `getServerSideProps`, `envInfo`, エラー処理
- **`index.tsx`**: `Template` + `uiParts` + `organisms` の組み立て

例: `src/pages/hotel/index.page.tsx`はSSR・レビューAPIを繋ぎ、`HotelTop`は同じフォルダ`.`からimportします。

SEO・initialのルートが増えた後も **「このファイルを開いたらSSRから見るか、UIから見るか」** をチーム全体が同じパターンで読めるようにするための選択です。

#### `containers`は？

`containers/`には予約入力・決済などの**少数のRedux connect**のみがあります。containerパターンの全面導入というより、**予約・決済ドメインに残ったReact-Reduxスタイル**に近いです。

#### Story 1 要約

> **背景:** 検索・予約・TOP・SEO URLが1つのrepoに共存しているが、Atomicの**サイズ**だけでは配置の基準が曖昧で、organismsにコンテンツUIまで混ざり、SEO移行で**同じ役割のUIの重複**が明らかになった。
> **選択:** Atomicをひっくり返すのではなく、**フロー基準の`uiParts`**と**`*.page.tsx`で入り口・実装の分離**。
> **意味:** 完璧なデザインシステムではなく、**移行・成長期にチームが合わせた「配置の規則」**。

### Story 2. SEOリポジトリの統合

#### 状況

- SEOページは**別のリポジトリ**で開発・デプロイされていましたが、実際のURLはすべて`https://***/hotel/...`として運営されていました。
- `assetPrefix: '/hotel'`、Express・`_app`の`/_next/data` → `/hotel/_next/data`パッチなどで**同じprefix**を前提としていました。
- インフラではpath別のupstream分岐で処理していました。

#### 問題

- ホテルAPI・リスト・価格・FAQなど**ロジックが2つのrepoに重複**。
- URLパターンの爆発: `/hotel/initial/`, `/hotel/japan/onsen/.../`, `/hotel/tokyo/bl_***/`, `/hotel/review/2/` …
- SEO/ホテルの2つのリポジトリで並行作業が発生した。

#### 解決

**1つのrepoに統合しよう** — 私が最初に提案し、移行作業も主導しました。

| 領域 | 役割 |
|------|------|
| `src/pages/hotel/...` | SEO・エリア・initial・reviewルート |
| `src/utils/seo/*` | リスト・FAQ・ページング・メタなどのデータ組み立て |
| `src/constants/SEO/` | リダイレクト・エリアの定数 |
| `src/components/uiParts/` | FAQ・エリア説明・Swiper・URLページングなど |
| `next.config.js` | SEO redirects / rewrites, Cache-Control |

**レンダリング:** ほとんどが`getServerSideProps`(SSR)。ISR(`revalidate`)は`pages/hotel/japan/[area]/`など

#### 1ページの組み立て例 (都道府県エリア)

- **organisms:** `Template`, `Breadcrumbs`, `SearchBoxHotelSeo` — 骨格・検索
- **uiParts:** `AreaDescriptionSection`, `FaqList`, `ItemList`, `HotelPaging` — 縦に積むコンテンツブロック

#### 「Atomicで分ければよかったのでは？」
移行の最優先は**URL・SSR・APIの重複排除**でした。SEOの`HotelItem`と検索の`HotelItem`は型・Redux連動が異なるためorganismsにすぐに入れるのは難しく、すでに`uiPart`に集めていたURL・props UIと同じ性格だったので同じレイヤーに統合しました。

### インフラ・DX

#### `next.config.js`
- `trailingSlash`, SEO redirects / rewrites
- HTML vs static Cache-Control
- `pageExtensions`でルートファイルの規則を固定

#### 品質ツール
- ESLint `import/order` — `actions` → `components` → … レイヤー順
- Stylelint + styled-components, Husky + lint-staged
- cypress

### OTAプロジェクト要約
- 1つのrepo内に検索・予約アプリとTOP・SEO・initial・reviewのようなコンテンツURLが共存していた。
- atoms~organismsはRedux・API中心のアプリ本体、uiPartsはprops・URL中心のセクションUI — Atomicのサイズではなくフローで分けた。
- SEO統合は単純なコードのmergeではなく、URL・SSR・重複API・コンポーネントの境界を合わせる作業だった。
- *.page.tsxはルート・データの入り口と画面の実装をファイル名で区別しようとする選択だった。

ホテルrepoを1枚で描くと、Next PagesとReduxで検索・予約を回すアプリの上に、同じ/hotelの下の様々なURLがくっついていった形です。server.jsとnext.config.jsはネームスペース・redirect・キャッシュのような運用前提を合わせ、src/ではアプリ本体とコンテンツ・URLページ用のUIをフォルダで分けました。
SEOを1つのrepoに集める際、その境界が特に必要になり、ルートが増えるにつれてpageExtensionsで入り口と実装を分けました。
これはAtomic Designを定石通りに完成させた事例というより、画面の種類が増える時に現在のチーム状況に合わせた実務的な対応に近いです。フォルダ名だけでは「グローバル状態が必要か」、「URLとpropsだけで描画すればいいのか」が見えなかったので、それをimport経路とページファイルの規則で明らかにするようにしました。

## ウェブ小説公募展サイト - VueからNext.jsへ、毎年変わるビジュアル

### このプロジェクトは？

| 項目 | 内容 |
|------|------|
| **サービス** | ウェブトゥーン・ウェブ小説プラットフォームのウェブ小説公募展サイト (ランディング + 応募フォーム) |
| **スタック** | Next.js 13 (App Router), React 18, Redux Toolkit, Tailwind CSS + SCSS, TypeScript |
| **ビルド・デプロイ** | `output: 'export'` 静的ビルド → CDNサービング |
| **役割** | Vue → Next.jsマイグレーション、新規年度の開発、アーキテクチャ設計を担当 |

### 背景

このプロジェクトは元々Vueで作られていました。担当エンジニアが退職した後、公開のみされている状態であり、会社はNext.jsを導入しようとする方向性がありましたが、社内にNext.jsをきちんと扱える人員がいませんでした。私がアサインされたのもその文脈です。

行ったことは2つです
1. 既存のVueコードをNext.jsへマイグレーション (昨年のイベントアーカイブ)
2. 新規年度のイベントページをNext.jsで新規開発

イベントページは本サービスとは独立しており規模が限定的で、かつ毎年新しく作成するため、新しいスタック導入のパイロットとして適した条件でした。社内にNext.jsの経験者がいない状態で私が初めて作るコードだったため、今後他のプロジェクトでこのrepoを参照する状況を前提としていました。App Router、Tailwind、フォルダ規則などの選択を**他のメンバーがこのコードを見て真似て作る時に自然な構造か**を基準に判断しました。

### Story 1. スタックの選択 — App RouterとTailwind

#### なぜApp Routerなのか

##### 1) page.tsxだけがルートになる
Pages Routerでは`pages/`の中のすべてのファイルが自動的にルートになります。ホテルプロジェクトではこのため`pageExtensions: ['page.tsx']`という迂回規則を作って、ルートファイルと画面組み立てファイルを区別しました。
App Routerでは **`page.tsx`という名前のファイルだけがルートとして扱われます。** 同じフォルダに他のファイルがあってもルートにはなりません。ホテルで設定によって解決した問題が、App Routerでは基本動作で解決されます。
**「ルートの入り口」と「画面の実装」の分離**が構造的に保証されます。

##### 2) Nested Layout — ページ遷移時にレイアウトが維持される
`layout.tsx`にReduxProvider、GTM、年度別のHeader/Footerを構成すると、**下位ページが遷移してもこのレイアウトは再レンダリングされません。**

```tsx title="例えばこんな感じ"
// layout.tsx
export default function RootLayout({ children }) {
  return (
    <html lang="ja">
      <GTM/>
      <body>
        <ReduxProvider>
          <Main>{children}</Main>   
        </ReduxProvider>
      </body>
    </html>
  )
}
```

応募フォームはregister → confirm → completeの3stepです。ページ遷移時にReduxProviderが維持されるので、registerで入力したフォームデータがconfirmで自然に保持されます。Pages Routerでも可能ですが、`_app.tsx`1つにすべてのラッパーを入れるか、`getLayout`パターンを別途作る必要があります。

##### 3) サーバーコンポーネント — page.tsxがクライアントJSを出力しない

App Routerで`'use client'`を宣言していないコンポーネントはサーバーコンポーネントであり、クライアントバンドルにJavaScriptを送りません。このプロジェクトのpage.tsxはすべてサーバーコンポーネントであり、実際のJSは`'use client'`が宣言されたコンポーネントから始まります。
`output: 'export'`(静的ビルド)においてサーバーコンポーネントはビルド時に実行されます。サーバーコンポーネントの複雑な部分(ストリーミングなど)は静的ビルドでは使用できないため悩む必要がなく、 **「ビルド時のレンダリング + クライアントJSの最小化」** というメリットだけを得ることができます。

##### 4) Metadata API
OGP・Twitter Cardの設定が`export const metadata`として宣言的に処理され、親layoutのmetadataが子pageに自動適用されます。Pages Routerの`next/head`の手動挿入よりも漏れにくくなります。

#### なぜTailwind CSSなのか
1) 毎年捨てるビジュアルにクラス名が必要か
このサイトは**毎年ビジュアルが完全に変わります。** 今年の`SeasonTheme`は来年にはアーカイブとなり、新しい`SeasonTheme`を作ります。
SCSSで作業すると、毎年1年だけ使うクラス名を定義し、別ファイルで管理する必要があります。Tailwindはマークアップにスタイルがくっついているため**コンポーネントファイル1つで完結**し、「昨年のフォルダを参考にして今年のフォルダを作る」ワークフローでスタイルファイルを漏らすことがありません。早い開発スピードもイベントページの特性に合っていました。

2) 静的ビルドでの効率
Tailwindは使用したクラスだけを残すpurgeが基本であり、ランタイムのJSオーバーヘッドがありません。`output: 'export'`で静的ファイルをCDNに上げる構造において、バンドルサイズをできるだけ減らすことがすなわちパフォーマンス向上に繋がると考えました。

3) 既存SCSSとの共存
2023年アーカイブのSCSS、フォームコンポーネントのSCSSはそのまま残し、**2024年の新規コンポーネントにのみTailwindを導入**しました。
```text
globals.css  → Tailwind (2024ホームセクション、共通ボタン)
main.scss    → SCSS (2023アーカイブ、フォーム、レイアウト)
```
結果的にSCSSは「変わらないもの」(フォーム、アーカイブ)、Tailwindは「毎年新しく作るもの」(ホームセクション)に自然に対応するようになりました。

### Story 2. 年度別アーキテクチャ — 「変わるもの」と「変わらないもの」

#### 問題
公募展は毎年開催されます。ビジュアルは完全に変わりますが構造は同じです:
- **ホームランディング** — シーズンテーマ、日程、応募要項、参考作品
- **応募フォーム** — 入力 → 確認 → 送信 → 完了

昨年のページをアーカイブとして残しつつ、今年のページを新しく作る必要があります。

### 選択: 年度フォルダの分岐

| 年度別に分かれるもの | 年度に関係なく共有されるもの |
|---|---|
| `contents/2023/Home.tsx` | `contents/Register.tsx` |
| `contents/2024/Home.tsx` | `contents/Confirm.tsx` |
| `layout/2023/footer.tsx` | `common/element/*` |
| `layout/2024/header.tsx` | `store/*`|
| `constants/2023/products.ts` | `features/apiRequest.ts` |

#### 新しい年度を追加する時にやること
1. `contents/{year}/`フォルダにホームセクションコンポーネントを新しく作成
2. `layout/{year}/`にその年のheader/footerを追加
3. `constants/{year}/`に作品データ・テーマデータを追加
4. `yearManager.tsx`に分岐を追加

**手を加えないもの:** フォームフロー、store、APIラッパー、共通コンポーネント。

#### `yearManager.tsx` — 年度分岐のハブ

```tsx
export const getYearlyHeaderAndFooter = (pathName: string) => {
  const yearMatch = pathName.match(/\/web\/(\d{4})/)
  const year = yearMatch ? Number(yearMatch[1]) : THIS_YEAR
  // pathnameから年度を抽出してheader/footerコンポーネントを返す
}
```

`main.tsx`ですべてのページを`{header()} {children} {footer()}`で囲むため、新しい年度を追加する時に`layout.tsx`は修正する必要がありません。

### ホテルプロジェクトと同じ観点で見ると

| 観点 | ホテル | イベントサイト |
|------|------|------------|
| **レンダリング** | SSR (Express + getServerSideProps) | 静的ビルド → CDN |
| **ルーター** | Pages Router + `pageExtensions`規則 | App Router (基本動作で解決) |
| **状態管理** | Redux + thunk + redux-form | Redux Toolkit (slice別のカプセル化) |
| **スタイル** | styled-components | Tailwind + SCSS混用 |
| **UI分類基準** | フロー (アプリ vs コンテンツ → organisms vs uiParts) | 年度 → `contents/{year}/` |
| **pageファイル** | 重い (SSRロジック) | 軽い (委譲のみ) |
| **プロジェクトの文脈** | 既存チームrepoに参加 | 単独での設計・実装 |

ホテルで **グローバル状態が必要か？** がorganisms/uiPartsの境界だったとすれば、ここでは **毎年変わるか？** が境界です。ドメインが違うので分類軸も違いますが、**役割が異なるUIはフォルダで分ける**という原則は同じです。

### 振り返ってみて - 改善できたこと

#### contentsフォルダの二重の役割
`components/contents/`の中に年度別ホーム(2023/, 2024/)と共有フォーム(Register.tsx, Confirm.tsx)が同じ階層にあります。ライフサイクルが異なるコードが1つのフォルダに共存しているわけです。フォームコンポーネントを`contents/apply/`や別のパスに分離していれば、「毎年変わるもの / 変わらないもの」の境界がフォルダ構造でより明確に見えたはずです。
例えばこんな風にしていれば、もっと良いコードになったのではないかと思います。 🤔
```text
components/contents/
├── 2023/              ← 毎年変わる
│   └── ...
├── 2024/              ← 毎年変わる
│   └── ...
└── apply/             ← 変わらない (共有) — フォルダでまとめる
    ├── Register.tsx
    ├── Confirm.tsx
    └── Complete.tsx
```
#### 静的ビルド環境でのトレードオフ
App Routerで一番気を使った部分は'use client'の境界でした。このプロジェクトはReduxとuseStateを多用するため、実際のUIコンポーネントのほとんどはクライアントコンポーネントになります。サーバーコンポーネントのバンドル最小化のメリットはpage.tsxレイヤーにのみ適用され、コンポーネントレベルでは大きく体感できませんでした。
ただし、これはこのプロジェクトが静的ビルド(output: 'export')であるため、サーバーコンポーネントの本当の強みであるサーバーサイドのデータフェッチングやストリーミングを使えなかったということもあります。SSRを使うプロジェクトであれば、サーバーコンポーネントのメリットがより大きかったはずです。
それでもpage.tsxのルート区別、nested layout、Metadata APIのような構造的なメリットは静的ビルドでも有効だったため、App Routerの選択自体は正解だったと思います。"

### ウェブ小説プロジェクト要約
- 社内にNext.jsの経験者がいない状態で、VueのイベントサイトをNext.jsへマイグレーションし、新規開発まで単独で進めました。今後他のプロジェクトの参照になるコードだという前提の下、スタックと構造を決定しました。
- App Routerはpage.tsxのルート区別、nested layout、サーバーコンポーネントによるバンドル最小化、Metadata APIがこのプロジェクトの構造(静的ビルド、軽いpage、3stepフォーム)に合っていました。
- Tailwindは毎年捨てるビジュアルにクラス名を作る必要がないという点、早い開発スピード、既存SCSSとの衝突のない共存が導入の根拠でした。
- ディレクトリの核心設計は **「毎年変わるもの vs 変わらないもの」の分離。** ホームセクション・ヘッダー・作品データは年度別に分けられ、フォームフロー・store・APIは共有されます。
- ホテルプロジェクトが「フロー基準の分類」を語ったとすれば、このプロジェクトは「年度基準の分類」と「パイロットプロジェクトでのスタック選択」と言えます。

2つのプロジェクトを並べて見ると、フォルダ規則には正解があるわけではなく、プロジェクトの核心目的とメンテナンス戦略によって決定されるということを感じます。ホテルプロジェクトでは **ドメインビジネスロジックとの結合度(アプリフロー vs コンテンツページ)** が基準でしたが、ウェブ小説プロジェクトでは **コードのライフサイクルと修正頻度(毎年変わるテーマ vs 固定された機能)** が基準でした。

## オファー型採用プラットフォーム — 守られていなかったアトミック構造をシンプルに変える

### プロジェクト概要

| 項目 | 内容 |
| --- | --- |
| **サービス** | 企業担当者向け採用スカウトプラットフォーム (B2B) |
| **スタック** | Next.js 14 (Pages Router), React 18, Apollo Client, React-Hook-Form, SCSS Modules, Storybook 8 |
| **役割** | Next.js 13 → 14バージョンアップグレード専任およびコンポーネント全数調査による大規模な構造リファクタリングを主導、Storybookインフラの再構築 |

### 背景
初めてプロジェクトに合流した時、Next.js 13 Pages Routerで運用中でした。ビジネス機能は稼働していましたが、チームメンバーの主要技術スタックはLaravel中心のバックエンド領域でした。そのためフロント側のコンポーネントレイヤーの明確な共通規則が不在で、技術的負債が徐々に蓄積している状況でした。
当時、フレームワークのパラダイム転換期であったため、App Routerへのマイグレーションも代案として検討できましたが、Laravel中心の組織の技術的連続性と工数リスクを考慮した際、全面改修は実用的なアプローチではないと判断しました。これによりアーキテクチャの目標を、フレームワークの転換ではなく、現在与えられたPages Router環境の欠陥を治癒し、正しく使うことに再定義しました。これを実現するために、Next.js 13から14へのバージョンアップグレード、エラーで麻痺していたStorybookインフラの再構築、そしてコンポーネント構造の全面的なリファクタリングを設計・実行しました。

### 発見した問題点 (技術的負債)
#### 1) 抜け殻だけ残っていたアトミックデザイン
プロジェクト内部には理論的な規格である`atoms/`, `molecules/`, `organisms/`ディレクトリが作成されていましたが、実際のコンポーネントの配置は全面的に開発者の主観的判断に依存していました。

#### 2) グローバルな共通コンポーネントとページ専用コンポーネントの混在
画面全体で再利用される共通UIコンポーネントと、特定のページでのみ1回限り使われるドメイン従属型コンポーネント(チャート、特定セクションのUIなど)が明確な分離基準なしに`components/`ディレクトリ内に混ざり合っていました。これにより特定のコードを修正する際の影響範囲がわかりにくい問題があり、全く使用されず放置されたコンポーネントもありました。

#### 3) 麻痺して放置された開発者体験(DX)インフラ
コンポーネント駆動開発(CDD)の核となるべきStorybookは、内部モジュールの依存関係エラーにより実行すらできない状態で放置されていました。

### 解決過程 - アトミックの解体とColocationパラダイムのバックポート

#### 1段階. 大々的なコンポーネント全数調査 (洗い出し)
単なるファイルの移動に先立ち、ソースコード全体を全数調査し、各コンポーネントの実際の依存関係とimport経路を把握しました。この過程で発見された、どのページからも呼び出されていない多数のコンポーネントを先に削除しました。

#### 2段階. アトミックレイヤーの廃棄と使用箇所基準のフラット化
曖昧なサイズ基準の分類体系を全面的に廃棄しました。グローバルで再利用される本当の共通UIコンポーネントは、レイヤーの段階をすべて取り払い、`components/`の直下に単一レベルでフラットに並べました。

#### 3段階. pageExtensionsを活用したColocationアーキテクチャの移植 (Backporting)
ウェブ小説プロジェクトでApp Routerを経験し、一番良かったと思ったのが **コードが使われる場所にコードを近く置くというColocation(近接配置)** でした。

すぐにフレームワークをApp Routerに変えることはできませんでしたが、この方法をPages Router環境でも実装したいと思いました。最初のホテルプロジェクトで、ルートファイルと画面組み立てファイルを区別するために`pageExtensions`設定をカスタムした経験を思い出しました。
この方法で`next.config.js`の設定を修正し、`*.page.tsx`のみをルーターの進入点として固定し、各ページフォルダ内部にルーターが収集しないプライベート空間である`_components/`領域を強制しました。

```text title="Pages Router環境に移植したColocation構造の例示"
pages/dashboard/
├── index.page.tsx            ← Next.jsエントリ (ルーティング、getServerSidePropsデータ制御)
└── _components/              ← ダッシュボード画面でのみライフサイクルを持つ専用コンポーネントの隔離空間
    ├── approach-chart.tsx
    ├── scout-status.tsx
    └── scout-status.stories.tsx
```
この画面でのみ使われる専用UIはページフォルダの配下に隔離し、今後機能を修正したり削除する際にすぐ見つけられるようにまとめました。

#### 4段階. Storybookの復活と3ファイルパッケージングコンベンションの定着
壊れていたStorybookはきれいに削除し、新しくインストールしてひとまず実行可能に直しました。そしてコンポーネントの修正・作成時に **コンポーネント(*.tsx) + スタイル(*.module.scss) + Storybook(*.stories.tsx)** の3つのファイルが1つのディレクトリ内でセットとして動くようチームに伝播しました。

### 主なレイヤー別の役割

| 領域 | 役割 |
| --- | --- |
| `*.page.tsx` | ルート定義、SSR(`getServerSideProps`)、画面組み立て |
| `pages/{route}/_components/` | 該当ページの**専用**コンポーネント — 他の場所からはimportしない |
| `components/` | 複数のページで再利用される**共通UI** — フラットに並べる |
| `hooks/` | 状態・イベント・API呼び出しを抽象化したカスタムフック |
| `services/` | GraphQL / REST APIラッパー |
| `contexts/` | 認証・グローバルUI状態などのReact Context |

### App Routerとの繋がり

ウェブ小説プロジェクトでApp Routerを使う時、最も体感した利便性の一つは**Colocation**でした。App Routerでは`page.tsx`のみがルートになり、同じフォルダの他のファイルは自動的に専用コンポーネントのように扱われます。ルートファイルの横にコンポーネントを置くことがフレームワークの基本動作でした。
このプロジェクトはPages Routerを維持しました。しかし`pageExtensions`と`_components/`規則で似たような構造を作りました。

後になってリサーチしながら、このような方式を「Colocation」と呼び、最新のUIライブラリも共通コンポーネントをこのようにフラットに管理していることを知りました。大げさな理論を知っていて適用したわけではなく、どうすれば現場の混乱を減らしてメンテナンスしやすくできるかを悩んだ方向が、幸いフロントエンドエコシステムのトレンドと大きく外れていなかったようで、非常に良かったと思っています。

## 3つのプロジェクトのアーキテクチャ整理

| 観点 | OTAプラットフォーム | ウェブ小説プラットフォーム | スカウトサービス |
| --- | --- | --- | --- |
| **フレームワーク** | Next.js 12 Pages Router | Next.js 13 App Router | Next.js 13 -> 14 Pages Router |
| **アーキテクチャのキーワード** | 制約条件との妥協 | 新技術のパイロット実験 | 複雑な規則の排除および構造の単純化 |
| **コンポーネント分類** | **フロー** (アプリロジック vs 静的コンテンツ) | **ライフサイクル** (シーズン性の揮発 vs 持続資産) | **使用箇所** (グローバルな共通 vs ページ専用の隔離) |
| **ルーティング統制方式** | `pageExtensions`カスタム設定 | App Router基本内蔵機能 | `pageExtensions` + `_components/` バックポート |
| **主たる核心成果** | 断片化したマルチリポジトリのアーキテクチャ統合 | 新規スタックの検証および複数年度のアーカイブ分離 | 技術的負債の解消およびDXインフラの再建 |

## おわりに

3つのプロジェクトを経て感じたのは、フォルダ規則やデザインパターンには当然ながら正解がないという点です。当時のサービスが状況とチームに合わせて構成するのが最善だと思います。

* **ホテルプロジェクト**では、画面の性格(複雑な機能が入るアプリなのか、単なる情報伝達用なのか)によってフォルダを分け、
* **ウェブ小説プロジェクト**では、コードのライフサイクル(応募が終わればアーカイブに転換される該当年度のビジュアルテーマなのか、来年も引き続き再利用される共通UIなのか)を基準にしました。
* **スカウトプロジェクト**では、アトミックのような理論的な規則よりも「このコンポーネントが実際にどこで使われているのか」という実用的な質問に集中しました。

結局のところ、フォルダ構造を悩む理由は、大げさなアーキテクチャを組むためではなく、一緒に働く人々が迷うことなく楽に開発できるようにするためだと考えています。フォルダを開いた時、他のチームメンバーに聞かなくても「あ、このコードはこういう理由でここにあるんだな」と直感的に理解できる構造が、結局のところ一番良い構造なのではないかと思います。
